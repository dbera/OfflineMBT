= Expression Functions Framework Design

== Overview

The Expression Functions Framework provides an extensible mechanism to register custom functions and value converters for use within expression evaluation. The framework automatically generates an Xtext-based expression DSL from registered function libraries and makes them available through a scoping provider using an in-memory resource set.

== Architecture

=== Core Components

The framework consists of the following key components that work together to provide function registration and discovery:

* <<ExpressionFunctionLibrariesProvider>> - Extension point for contributing custom function libraries
* <<ExpressionFunctionsRegistry>> - Central registry managing registered function libraries
* <<LibraryToExprGenerator>> - Auto-generates Xtext grammar from function libraries
* <<InMemoryExprResourceRegistry>> - Stores generated artifacts in-memory
* <<ExpressionsImportUriGlobalScopeProvider>> - Makes functions available through scoping
* <<DefaultExpressionFunctions>> - Default built-in function implementations

== Component Details

=== ExpressionFunctionsRegistry

**Purpose**: Central registry that manages all registered function libraries, validates function methods, and invokes functions with proper type conversion and argument validation.

**Responsibilities**:

- Registers function libraries from providers at initialization
- Validates methods before registration (visibility, return type, templatization)
- Maintains a mapping of function names to their method implementations
- Finds matching overloads based on parameter count and type compatibility
- Invokes functions with automatic type conversion using converters

**Method Registration Validation**:

During registration, each method is validated against the following constraints:

- **Public visibility**: Only `public` methods are registered; non-public methods are ignored
- **Non-void return types**: Methods must not return `void` or `Void` 
- **No Object return type**: Methods must not return `Object` (java.lang.Object), as the return type would map to "any" — too ambiguous for expressions
- **Templatization check**: Methods are templatizable only if they return a concrete, mappable type (not void/Void/Object)

Methods that violate these constraints are skipped during registration, and an error is logged. This ensures only well-defined functions are available in expressions.

**Method Overloading**:

Multiple methods with the same name but different parameter lists are supported. Overload resolution occurs at invocation time by matching:
- Parameter count (arity)
- Argument type compatibility with each parameter position

The first matching overload is selected and invoked.

**Key Methods**:

- `getFunctions()` - Returns Map<String, List<Method>> of all registered functions
- `hasFunction(String name)` - Checks if a function with the given name exists
- `invokeFunction(ExpressionFunctionCall functionCall, IEvaluationContext context)` - Finds matching overload, converts arguments, invokes method, and returns result as an Expression

**Integration**:

- Uses `IExpressionConverter` instances for bidirectional type conversion
- Coordinates with `InMemoryExprResourceRegistry` for storing generated function definitions
- Receives library classes from `IExpressionFunctionLibrariesProvider` at initialization

=== ExpressionFunctionLibrariesProvider

**Purpose**: Extension point for contributing custom function libraries.

**Pattern**: Returns the set of classes to be registered. To be specified in the (derived) language runtime.

**Responsibilities**:

- Enable dynamic library discovery at runtime

**Key Methods**:

- `get()` - Return a collection of function libraries as classes

=== Function Library Class

**Pattern**: Plain Java classes with public methods.

**Characteristics**:

- Each method represents an individual function
- Function overloading allowed

See also <<how-to-add-an-expression-function-library>>

=== LibraryToExprGenerator

**Purpose**: Automatically generates Xtext expression grammar from registered Java function libraries.

**Responsibilities**:

- Introspects Java method signatures
- Generates corresponding Xtext expression function definitions
- Creates in-memory _expr_ Xtext models
- Registers generated artifacts with the registry

**Key Features**:

- Automatic type mapping (Java types → Xtext types)
- **Template function generation**: Methods using generated DSL expression collection types (`ExpressionVector`, `ExpressionMap` — defined in Expressions.xtext) automatically generate generic/template function declarations

**Template Synthesis**:

When a Java method uses the generated DSL expression collection types in its signature, `LibraryToExprGenerator` synthesizes type parameters to create a generic template function:

- **Vector context** (presence of `ExpressionVector`): `ExpressionVector` → `T[]`, `Expression` parameter/return → `T`
  - Example: `Expression get(ExpressionVector v, long i)` generates `function <T> T get(T[] p0, int p1)`

- **Map context** (presence of `ExpressionMap`): `ExpressionMap` → `map<K, V>`, `Expression` as parameter → `K`, `Expression` as return → `V`
  - Example: `Expression get(ExpressionMap m, Expression key)` generates `function <K, V> V get(map<K, V> p0, K p1)`
  - Example: `boolean hasKey(ExpressionMap m, Expression key)` generates `function <K, V> bool hasKey(map<K, V> p0, K p1)`

This design allows a single Java method to work generically across different element types in vectors and maps without requiring separate overloaded methods in Java. Methods using only concrete types (not collections) map directly to their corresponding expression types (e.g., `String` → `string`, `long` → `int`).

=== InMemoryExprResourceRegistry

**Purpose**: Manages in-memory storage of the generated expression artifacts.

**Responsibilities**:

- Stores generated Xtext _expr_ models in memory
- Provides resource lookup by URI
- Maintains lifecycle of generated resources

**Key Methods**:

- `addLibrary()` - Register an in-memory resource
- `getURIHandler()` - returns the shared URI handler that can be registered in a resource set

=== ExpressionsImportUriGlobalScopeProvider

**Purpose**: Makes registered functions available to expression scoping through URI-based imports.

**Responsibilities**:

- Extends `TypesImportUriGlobalScopeProvider` class
- Resolves import URIs to function libraries
- Provides scope visibility for in memory expressions via imports

=== DefaultExpressionFunctions

**Purpose**: Makes the default built-in functions available in all languages derived from expressions. This way they can be used in expression evaluations.

**Built-in Functions**: isEmpty, size, contains, add, asReal, abs, hasKey, deleteKey, get, at, toString, concat, range

**Integration**: Automatically registered in `ExpressionFunctionsRegistry` during initialization via `ExpressionFunctionLibrariesProvider`.

=== ExpressionConvertersProvider

**Purpose**: Extension point for contributing custom value converters.

**Pattern**: Returns the set of converters to be used for conversion. To be specified in the (derived) language runtime.

**Responsibilities**:

- Define custom type conversion logic
- Support for bidirectional conversions
- Integration with expression evaluation engine

**Key Methods**:

- `get()` - Return a collection of converter implementations

== Dynamic Expression DSL Generation

When a library is registered, `LibraryToExprGenerator` introspects its method signatures, maps Java types to expression types, and produces an in-memory `.expr` resource stored in `InMemoryExprResourceRegistry`. `ExpressionsImportUriGlobalScopeProvider` adds these URIs to every resource set so the generated function declarations are visible without an explicit `import` — enabling content assist, validation, and navigation in the IDE.

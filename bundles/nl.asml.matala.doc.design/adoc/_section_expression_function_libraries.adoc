== Expression Function Libraries

The expression language supports user-defined function libraries.
A library is a plain Java class whose `public` methods (static or instance) are automatically exposed as callable functions in `.expr` models via the `call` keyword.

=== How to add a library

. Create a Java class with `public` methods — static or instance:
+
[source,java]
----
public class MyFunctions {
    // Static: no instance needed
    public static long square(long value) {
        return value * value;
    }

    // Instance: requires a receiver object at evaluation time
    public String format(String template, long value) {
        return String.format(template, value);
    }
}
----

. Register it with the `ExpressionFunctionsRegistry` (typically in your runtime module or setup code):
+
[source,java]
----
@Inject ExpressionFunctionsRegistry registry;

registry.addLibraryFunctions(MyFunctions.class);
----

. The functions are now available in any `.expr` model:
+
[source]
----
int result   = call square(5)
string label = call format("%d items", 3)
----

When `addLibraryFunctions` is called, two things happen automatically:

* The Java methods are registered for invocation at evaluation time.
* An in-memory `.expr` resource containing the `function` declarations is generated via `LibraryToExprGenerator` and registered in the `InMemoryExprResourceRegistry`.
  The `ExpressionsImportUriGlobalScopeProvider` adds these URIs to every resource set, so the declarations are visible without any explicit `import`.

=== Providing instances for non-static methods

Static methods are invoked directly. For instance methods, a receiver object must be provided at evaluation time via `IEvaluationContext.getLibraryFunctionObjects()`.

Override this method in your evaluation context to return the relevant instances:

[source,java]
----
IEvaluationContext ctx = new IEvaluationContext() {
    private final MyFunctions myLib = new MyFunctions();

    @Override
    public Expression getExpression(Variable variable) { ... }

    @Override
    public List<Object> getLibraryFunctionObjects() {
        return List.of(myLib);
    }
};
----

At invocation time, `ExpressionFunctionsRegistry` searches the list for the first object whose class is assignable to the method's declaring class and uses it as the receiver.
If no matching instance is found for a non-static method, an `IllegalStateException` is thrown.

=== Library class constraints

Methods exposed as functions must be::
  `public` — non-public methods are ignored.
  Both `static` and instance methods are supported.

Return type must not be `void`::
  Void methods are skipped during generation because expression functions must produce a value.

Supported parameter and return types::
+
[cols="1,1", options="header"]
|===
| Java type | Expression type

| `boolean` / `Boolean`
| `bool`

| `long` / `Long` / `int` / `Integer` / `short` / `byte`
| `int`

| `double` / `Double` / `float` / `Float` / `BigDecimal`
| `real`

| `String`
| `string`

| `List<T>` / `Collection<T>`
| `T[]` (vector)

| `Map<K, V>`
| `map<K, V>`

| `Object` / generic type variable
| `any`
|===

Overloaded methods are supported::
  Multiple methods with the same name but different parameter types or counts are allowed (e.g. `abs(long)` and `abs(BigDecimal)`).
  Overload resolution is based on the expression argument types at the call site.

Avoid generics with unchecked casts::
  Prefer concrete overloads over a single generic `<T>` method.
  For example, split `abs` into `abs(long)` and `abs(BigDecimal)` instead of using `<T> T abs(T value)`.

Parameter names::
  If the library is compiled with the `-parameters` flag (configured in both `pom.xml` and the Eclipse setup), the real parameter names appear in the generated `.expr` declarations.
  Otherwise, synthetic names (`p0`, `p1`, …) are used.

=== Architecture overview

The following components collaborate to make library functions available:

`LibraryToExprGenerator`::
  Reflects on the library class and generates the `.expr` function declarations as a string.

`InMemoryExprResourceRegistry`::
  Stores the generated content under a synthetic `inmemory:/expr/<class>.expr` URI and provides a `URIHandler` so the Xtext resource set can load it.

`ExpressionFunctionsRegistry`::
  Central registry that holds the Java `Method` references for invocation and delegates to `InMemoryExprResourceRegistry` for scope registration.

`IEvaluationContext`::
  Carries the library object instances used as receivers for non-static methods.
  Implementations override `getLibraryFunctionObjects()` to supply the relevant objects.

`ExpressionsImportUriGlobalScopeProvider`::
  Installs the in-memory URI handler on the resource set and adds all registered URIs to the global scope, making the function declarations visible to all `.expr` models without explicit imports.

`DefaultExpressionsConverter`::
  Handles bidirectional conversion between expression model objects (`ExpressionConstantInt`, `ExpressionVector`, `ExpressionMap`, …) and Java types (`Long`, `List`, `Map`, …) during function invocation.
